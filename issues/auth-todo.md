# Shared Authentication Decision Queue

Status: **an experimental shared Biei/Ishikari slice is implemented but not enabled in the demo.** Biei protects static renders and Ishikari protects public delivery content when explicitly configured. Both use one object-store `current.json` per registry and accept either Bearer or `access_token` query transport while rejecting mixed/repeated credentials. Ishikari propagates only verified query credentials into URLs it generates itself; it neither modifies retained external style URLs nor converts headers into URL credentials. The broader proposal remains exploratory in [`../specs/auth-sketch.md`](../specs/auth-sketch.md). This file owns unresolved decisions; it is not a roadmap and does not duplicate service-specific work.

## Settled constraints

- Verify steady-state requests locally from an immutable in-memory snapshot; never query storage per token or repeat a control-plane lookup for every request. One bounded, single-flight cold registry activation is the only planned request-adjacent exception.
- If dynamic delivery credentials are adopted, use one conditionally replaced, self-contained `current.json` per configured registry under its trusted auth root. It contains the complete token verifier set and namespace/action grants for that registry; runtime readers never list a prefix, infer latest state from object names, or fetch one object per token.
- Prefer a dedicated registry bucket/container and workload identity per environment. A prefix is organization, not a portable authorization boundary; do not reuse Ishikari's content-reader identity.
- Store metadata, public verification material, high-entropy API-key verifier digests, and secret references—not raw API keys, HMAC masters, cap subkeys, or private signing keys.
- Prefer a conditional-write admin CLI/object writer before considering a public management API. Management uses cloud/workload identity; application credentials never authorize registry mutation.
- Authentication does not replace edge tenant/IP request limits. Origin-local limits cannot govern CDN cache-hit egress.
- Carry a bounded public `registry_id` in the opaque delivery key and resolve it only through a trusted local `registry_id -> auth_root` catalog. Unknown IDs cause no storage I/O. The token entry owns namespace/action grants; there is initially no namespace-to-registry allowlist or registry-level namespace ceiling because registry writing remains centrally trusted. For built-in credentials, authentication and all registry-derived policy use one captured registry revision; external AuthN may use separate immutable verifier state, but AuthZ still uses one policy snapshot.
- Keep `mmpf-auth` limited to the verifier, registry reader, and credential-carrier contract now consumed by both servers. Object-store content resolution, HTTP response policy, peer routing, and rate limiting stay with their owning services.
- Use one ordinary delivery-token registry and format in Biei and Ishikari. The coarse initial policy is `render.static` plus namespace-scoped `read`; `read` deliberately does not split basemap, terrain, glyph, sprite, or derived resources into separate permission kinds.
- Authorize the canonical resource identity consumed by the route handler, including equivalent percent-encoded path spellings; never scope a token against a raw URI segment that the handler later decodes to another namespace.
- A Biei workload identity may authenticate the service but must not widen the caller's namespace grants. Caller-selected resources use the ordinary token, or a capability attenuated from it; service-token-only reads are not sufficient authorization.
- A protected rendered-cache hit requires `required_namespaces ⊆ caller.readable_namespaces`. Biei now enforces this using the producer's complete normalized grant set as a conservative requirement upper bound. The render wire carries those grants, a domain-separated one-way credential-and-registry-revision cache partition, and the bounded redacted ordinary provider token selected for the experimental trusted-service model. The raw token is never a cache identity and is absent from rendered-output requirements, gossip, outcomes, metrics, and logs. The partition changes on policy revision and isolates positive/negative style and TileJSON caches, their single-flight loads, and worker-local native style state. Protected and unprotected outputs do not cross, and an independently completed weaker render cannot relax a resident entry. Biei attaches the token only to the configured exact provider origin. A deployment that needs node confidentiality or workload identity supplies mTLS at the service-mesh layer rather than changing the application protocol. Component contracts and the production-container E2E now prove actual profile-request propagation, full token-bearing FileSource cache/single-flight separation, Ishikari authorization before its shared resource cache, and broad-token/weak-token non-interference across a real native render. The remaining cache work is Ishikari's trusted, bounded style dependency namespace set.
- Continue with registry tooling and measured deployment evidence before capability URLs or additional verifier hierarchies.

## Distribution and propagation

The conditionally replaced registry `current.json` is the durable linearization point, so pods never reach consensus *among themselves* about registry state. Each independently installs only a completely validated registry snapshot. The detailed cache, peer, gossip, and internal-refresh contract lives in [`../specs/auth-sketch.md`](../specs/auth-sketch.md#10-configuration-and-registry-distribution); the remaining implementation gates are:

- **Baseline implemented in both servers: local cache plus conditional refresh.** Each pod verifies from an immutable in-process registry snapshot. Loading and refresh are single-flight and use the configured object's ETag. One bounded cold registry load is allowed; no request fetches a per-token object. Backend-specific validator behavior still requires the capability test below before production-like use.
- **Raft (and any pod-to-pod consensus) is rejected.** It would create a *second* authority to reconcile with object storage, and its stable-quorum requirement fights the deployment model (HPA-autoscaled 2–6 pods on Spot with frequent preemption → constant membership churn, election storms), while coupling auth availability to consensus health — the correlated-outage SPOF the design avoids. Raft earns its keep only when the replicas themselves *are* the durable authority with linearizable writes; here writes are out-of-band and pods are pure readers.
- **Gossip is an optional bounded advisory hint, never authority.** If added, it carries at most one overwritable latest-update tuple per node, not one persistent key per registry and never credentials, policy bodies, or auth-root URLs. Polling remains the correctness backstop; future periodic refresh and hint handling must be jittered and coalesced so they cannot stampede the store.
- **Peer-first loading requires content authority.** HRW may choose peers to collapse cold loads, but an unsigned peer snapshot cannot be installed solely because it came from the internal network. Either verify a management-plane signature over the exact snapshot or independently anchor its digest/generation in object storage; otherwise peer fetch saves bytes but not the authoritative read. Object storage remains the fallback.
- **Internal refresh is a hint.** A bounded internal endpoint may schedule an idempotent registry refresh, but it accepts neither auth-root URLs nor policy bodies and is protected against storage/CPU amplification by trusted-catalog checks, cooldowns, single-flight, and the internal-listener network boundary.
- **Read scopes have a loose revocation SLA, so gossip is an accelerator rather than a correctness requirement.** Read-only access to commodity map data has a low cost-of-staleness, and the real abuse ceiling is edge request/egress limits, not registry-propagation speed (see the edge-limit constraint above) — so a revocation SLA of minutes and relaxed polling remain sufficient when gossip is unavailable. Add gossip propagation when measured polling cost or the chosen revocation target justifies it. This biases the **read tier toward fail-open** on prolonged refresh failure (serve the last-good snapshot with a generous maximum age). Fail-closed is reserved for the private-data strong tier as a conscious per-scope choice (refines "Registry freshness and failure policy" below).

## Expansion and deployment gate

Before enabling the experimental slice in the demo or expanding it to more routes, explicitly decide that built-in authentication is a product requirement rather than relying only on a customer gateway, `TrustedHeader`, or service mesh. Name the operator responsible for key issuance, rotation, revocation, and incident response.

**Recommendation:** finish proving the object-store credential behavior on Biei's expensive `static` route and authenticated entry documents first. Do not start with Ishikari capability URLs while the CDN abuse boundary is unresolved.

## Required before production-like registry or strong-auth use

1. **Registry freshness and failure policy:** choose refresh interval, maximum last-known-good age, startup behavior without a valid snapshot, and when prolonged refresh failure becomes fail-closed. The interval is the normal revocation-latency SLA.
2. **IAM, signing, and secret backend:** choose a versioned external secret backend where symmetric keys or registry-signing keys are unavoidable; define separate registry-reader, writer, verifier-secret-reader, and issuer identities, rotation overlap, audit ownership, and the dedicated production bucket/container policy. Secret references must pin immutable provider versions retained through the registry rollback window; encrypted key blobs and private signing keys are not stored in registry objects. Decide whether peer distribution is deferred or management-signed from its first release.
3. **Promote or replace the experimental v1 schema:** review the implemented bounds for `registry_id`, monotonic revision, credential digest, enabled state, namespace/`render.static`/`read` grants, and exact Origin policy. Add only fields justified by the next protected route. Specify backend-generation CAS for writer replacement and the signed envelope only if peer distribution is enabled. The first schema has no namespace-to-registry allowlist or registry-level namespace ceiling; adding delegated registry writers requires revisiting that trust decision first.
4. **Backend capability matrix:** test which `object_store` backends support strong validators, conditional reads, version/generation-conditional replacement, and overwrite protection. Define how the CLI refuses unsafe multi-writer operation where those guarantees are unavailable; IAM alone does not require callers to use preconditions. Define current-object version/secret-version retention and garbage collection without deleting required rollback material.
5. **Credential contract:** validate Bearer and `access_token` behavior across supported clients and CDN/Gateway configurations; retain the bounded `<registry_id>.<opaque_registry_credential>` envelope, first-dot split, and rejection of mixed/repeated transports. Define per-adapter entropy or signature requirements, verifier construction, rotation overlap, one-time secret display where applicable, URL/log/referrer redaction, and error behavior. Do not store recoverable raw API keys merely because the registry is private.
6. **Dependency closure and resource isolation:** retain the coarse exact actions `render.static` and `read`, with `read` constrained only by bounded namespaces in the ordinary tier. The conservative Biei output-cache check, credential-partitioned profile/native-style state, bounded redacted token wire field, exact-provider propagation, FileSource credential separation, Ishikari auth-before-cache boundary, and composed native-render non-interference test are implemented. Specify the trusted wire shape and bounds for Ishikari's style dependency namespace descriptor and the policy/style revisions that invalidate it. Do not add basemap/terrain/resource-kind scopes without a concrete isolation requirement, and never derive grants from bucket names or physical prefixes.
7. **Registry expiry and rollback safety:** define forward-only rollback that preserves intervening revocation tombstones and requires separately audited explicit reactivation. Decide whether the registry snapshot carries signed `not_before`/`expires_at`, clock-skew bounds, and break-glass behavior. Monotonic numbering protects a running pod from regression but does not give a fresh pod anti-replay; decide whether object-store IAM/audit is sufficient or an external monotonic/transparency anchor is required.
8. **Shared-boundary review:** use deployment evidence from both servers to confirm that the extracted verifier surface is sufficient; do not move service routing, response policy, or content storage into it merely because authentication is cross-cutting.
9. **Composed cache non-interference test (implemented):** CI runs authenticated production Biei and Ishikari images against a real native render. A broad token warms Ishikari's resource cache and Biei's rendered-output cache; a weaker token that may read the style but not its tileset must neither read the cached tile nor receive the broad render, and the denied attempt must not poison the authorized cache entry. Keep this regression when either server, FileSource, the wire contract, or the auth crate changes. If exact-byte adaptive relaxation is later implemented, additionally prove the weaker render used no broader-authority cache entries and keep alternative requirement sets bounded.
10. **Credential-bearing service transport:** document whether the deployment accepts its internal network as the trusted bearer-token boundary. If it requires confidentiality or authenticated workload identity, prove mesh mTLS or an equivalent deployment-layer mechanism for Biei peer forwarding and Biei-to-Ishikari requests. Where identity authorization is enabled, admit only the intended Biei workload to `/_internal/forward`; split metrics onto a separate listener only if the selected mesh cannot express that distinction cleanly.

**Sequencing note.** A shared reader-only Biei/Ishikari prototype now exists, but the backend-capability check (#4) remains the first task before adding writer tooling or enabling the feature in a production-like deployment. A short spike must prove that generation-conditional replacement of registry `current.json` rejects a stale writer and that conditional reads return a trustworthy unchanged result on the *actual* production backend (GCS). If it fails, the registry-current model still needs rework; the disabled prototype is not evidence that the writer contract is safe.

## Deferred capability/CDN decisions

These block capability implementation but do not block registry tooling, strong
entry-route authentication, or the selected ordinary-token service path:

- Exact canonical payload and bounded parser for `version`, audience, `kid`, `key_id`, epoch, and policy revision; MAC length must provide at least 128 effective bits.
- Capability signing topology: per-key epoch subkeys versus a shared epoch key versus asymmetric signatures. A verifier pod that loads many symmetric keys can forge for every loaded key during the live epochs; per-key derivation does not make whole-pod compromise customer-local.
- Web and native-client epoch lengths. Accepting current and previous epochs permits a capability minted near an epoch start to live for almost two epochs.
- CDN cache-key behavior for the full capability URL and `Origin`, including absent/null/non-origin-form handling and 4xx caching.
- CDN-side request/egress controls, usage alerts, and kill switches. Per-pod token buckets protect only origin traffic and vary with replica count.
- Capability-bearing URL retention and redaction in CDN, Gateway, browser/client, support, and analytics logs.
- Emergency revocation semantics across registry refresh, origin snapshots, accepted epoch overlap, CDN expiry, and explicit purge.
- Whether capability rewriting covers third-party provider URLs or only self-hosted resources.
- Private-data policy: which scopes are always `no-store` and remain on the strong-auth path.
- Offline/bulk export contract; do not turn long-lived tile capabilities into an unbounded download API.

## Revisit object storage only with evidence

A database, public issuance API, global token registry, or push-invalidation subsystem is deferred. Reconsider only if per-registry current-object contention, measured registry size/decode cost, audit/query needs, secret fan-out, or the measured revocation SLA cannot be met by complete registry snapshots, conditional refresh, bounded gossip hints, and optional peer-assisted loading.
